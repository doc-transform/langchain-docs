多 Agent 系统通过协调专门化的组件来处理复杂的工作流。然而，并非每个复杂任务都需要这种方式 —— 一个拥有合适（有时是动态的）工具和提示词的单一 Agent 通常也能达到类似效果。

## 为什么需要多 Agent？

当开发者说他们需要"多 Agent"时，通常是在寻找以下一种或多种能力：

- <Icon icon="brain" /> **上下文管理**：提供专门化的知识，同时不使模型的上下文窗口过载。如果上下文是无限的且延迟为零，你可以将所有知识塞进一个提示词 —— 但现实不是这样，所以你需要各种模式来有选择地呈现相关信息。
- <Icon icon="users" /> **分布式开发**：允许不同团队独立开发和维护各自的能力，通过清晰的边界将它们组合成更大的系统。
- <Icon icon="git-branch" /> **并行化**：为子任务生成专门化的工作者，并发执行以获得更快的结果。

多 Agent 模式在以下场景特别有价值：单个 Agent 拥有太多[工具](/oss/langchain/tools)而无法做出正确的工具选择决策时；任务需要大量上下文的专门知识（长提示词和领域特定工具）时；或者你需要强制执行顺序约束，仅在满足特定条件后才解锁某些能力时。

<Tip>
多 Agent 设计的核心是**[上下文工程](/oss/langchain/context-engineering)** —— 决定每个 Agent 能看到什么信息。系统的质量取决于确保每个 Agent 都能访问到其任务所需的正确数据。
</Tip>

## 模式

以下是构建多 Agent 系统的主要模式，每种模式适用于不同的场景：

| 模式 | 工作方式 |
|------|----------|
| [**子 Agent**](/oss/langchain/multi-agent/subagents) | 主 Agent 将子 Agent 作为工具进行协调。所有路由都通过主 Agent，由它决定何时以及如何调用每个子 Agent。 |
| [**交接**](/oss/langchain/multi-agent/handoffs) | 行为根据状态动态变化。工具调用会更新状态变量，触发路由或配置变更，切换 Agent 或调整当前 Agent 的工具和提示词。 |
| [**技能**](/oss/langchain/multi-agent/skills) | 按需加载专门化的提示词和知识。单一 Agent 保持控制权，同时根据需要从技能中加载上下文。 |
| [**路由器**](/oss/langchain/multi-agent/router) | 路由步骤对输入进行分类，并将其定向到一个或多个专门化的 Agent。结果被综合成一个组合响应。 |
| [**自定义工作流**](/oss/langchain/multi-agent/custom-workflow) | 使用 [LangGraph](/oss/langgraph/overview) 构建定制的执行流程，混合确定性逻辑和 Agent 行为。将其他模式作为节点嵌入到工作流中。 |

| [**技能**](/oss/langchain/multi-agent/skills) | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

- **分布式开发**：不同团队能否独立维护各自的组件？
- **并行化**：多个 Agent 能否并发执行？
- **多跳**：模式是否支持串行调用多个子 Agent？
- **直接用户交互**：子 Agent 能否直接与用户对话？

<Tip>
    你可以混合使用这些模式！例如，**子 Agent** 架构可以调用那些会触发自定义工作流或路由器 Agent 的工具。子 Agent 甚至可以使用**技能**模式来按需加载上下文。可能性是无限的！
</Tip>

## 性能对比

不同模式具有不同的性能特征。理解这些权衡有助于你根据延迟和成本需求选择合适的模式。

**关键指标：**
- **模型调用次数**：LLM 调用的次数。调用越多 = 延迟越高（尤其是顺序执行时）且每次请求的 API 成本越高。
- **处理的 token 数**：所有调用中[上下文窗口](/oss/langchain/context-engineering)的总使用量。token 越多 = 处理成本越高且可能触及上下文限制。

### 重复请求

> **第 1 轮：**「买咖啡」
> **第 2 轮：**「再买一杯咖啡」

用户在同一对话中重复相同请求。

<div className="compact-first-col">

| 模式 | 第 2 轮调用次数 | 总计（两轮） | 最佳选择 |
|------|:--------------:|:----------:|:-------:|
| [**子 Agent**](/oss/langchain/multi-agent/subagents) | 4 | 8 | |
| [**交接**](/oss/langchain/multi-agent/handoffs) | 2 | 5 | ✅ |
| [**技能**](/oss/langchain/multi-agent/skills) | 2 | 5 | ✅ |
| [**路由器**](/oss/langchain/multi-agent/router) | 3 | 6 | |

</div>

<Tabs>
  <Tab title="子 Agent">
    **再次 4 次调用 → 总计 8 次**

    - 子 Agent **设计上是无状态的** —— 每次调用都遵循相同的流程
    - 主 Agent 维护对话上下文，但子 Agent 每次都从零开始
    - 这提供了强大的上下文隔离，但会重复完整的流程
  </Tab>

  <Tab title="交接">
    **2 次调用 → 总计 5 次**

    - 咖啡 Agent 从第 1 轮起**仍然活跃**（状态持久化）
    - 无需交接 —— Agent 直接调用 `buy_coffee` 工具（第 1 次调用）
    - Agent 回复用户（第 2 次调用）
    - **通过跳过交接步骤节省了 1 次调用**
  </Tab>

  <Tab title="技能">
    **2 次调用 → 总计 5 次**

    - 技能上下文**已经加载**在对话历史中
    - 无需重新加载 —— Agent 直接调用 `buy_coffee` 工具（第 1 次调用）
    - Agent 回复用户（第 2 次调用）
    - **通过复用已加载的技能节省了 1 次调用**
  </Tab>

  <Tab title="路由器">
    **再次 3 次调用 → 总计 6 次**

    - 路由器是**无状态的** —— 每次请求都需要一次 LLM 路由调用
    - 第 2 轮：路由器 LLM 调用 (1) → 咖啡 Agent 调用 buy_coffee (2) → 咖啡 Agent 响应 (3)
    - 可以通过封装为有状态 Agent 中的工具来优化
  </Tab>
</Tabs>

**关键洞察：** 有状态模式（交接、技能）在重复请求中节省 40-50% 的调用。子 Agent 保持每次请求一致的成本 —— 这种无状态设计提供了强大的上下文隔离，但代价是重复的模型调用。

### 多领域请求

> **用户：**「比较 Python、JavaScript 和 Rust 在 Web 开发方面的表现」

每个语言 Agent/技能包含约 2000 token 的文档。所有模式都可以进行并行工具调用。

| 模式 | 模型调用次数 | 总 token 数 | 最佳选择 |
|------|:----------:|:----------:|:-------:|
| [**子 Agent**](/oss/langchain/multi-agent/subagents) | 5 | ~9K | ✅ |
| [**交接**](/oss/langchain/multi-agent/handoffs) | 7+ | ~14K+ | |
| [**技能**](/oss/langchain/multi-agent/skills) | 3 | ~15K | |
| [**路由器**](/oss/langchain/multi-agent/router) | 5 | ~9K | ✅ |

<Tabs>
  <Tab title="子 Agent">
    **5 次调用，~9K token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-subagents.png" alt="子 Agent 多领域：5 次调用，并行执行" />
    </Frame>

    每个子 Agent 在**隔离**环境中仅处理其相关上下文。总计：**9K token**。
  </Tab>

  <Tab title="交接">
    **7+ 次调用，~14K+ token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-handoffs.png" alt="交接多领域：7+ 次顺序调用" />
    </Frame>

    交接模式**顺序执行** —— 无法同时研究三种语言。增长的对话历史会带来额外开销。总计：**~14K+ token**。
  </Tab>

  <Tab title="技能">
    **3 次调用，~15K token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-skills.png" alt="技能多领域：3 次调用，上下文累积" />
    </Frame>

    加载后，**每次后续调用都要处理全部 6K token 的技能文档**。由于上下文隔离，子 Agent 总体处理的 token 少 67%。总计：**15K token**。
  </Tab>

  <Tab title="路由器">
    **5 次调用，~9K token**

    <Frame>
      <img src="/oss/langchain/multi-agent/images/multidomain-router.png" alt="路由器多领域：5 次调用，并行执行" />
    </Frame>

    路由器使用 **LLM 进行路由**，然后并行调用 Agent。类似于子 Agent，但有一个显式的路由步骤。总计：**9K token**。
  </Tab>
</Tabs>

**关键洞察：** 对于多领域任务，支持并行执行的模式（子 Agent、路由器）效率最高。技能模式调用次数更少，但由于上下文累积导致 token 使用量较高。交接模式在此场景效率较低 —— 它必须顺序执行，无法利用并行工具调用来同时查询多个领域。
