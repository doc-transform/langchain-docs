## 概述

构建 Agent（或任何 LLM 应用）最困难的部分在于使其足够可靠。原型阶段也许运行良好，但在真实场景中往往会出现问题。

### 为什么 Agent 会失败？

当 Agent 出现故障时，通常是因为内部的 LLM 调用执行了错误的操作，或没有按预期工作。LLM 失败的原因通常不外乎两点：

1. 底层 LLM 的能力不足
2. 没有向 LLM 传递「正确」的上下文

在大多数情况下，真正导致 Agent 不够可靠的其实是第二个原因。

**上下文工程（Context Engineering）** 是指以正确的格式向 LLM 提供恰当的信息和工具，使其能够完成任务。这是 AI 工程师最核心的工作。缺乏「正确」的上下文是阻碍 Agent 可靠性提升的首要因素，而 LangChain 的 Agent 抽象正是为了便捷地实现上下文工程而专门设计的。

<Tip>
初次接触上下文工程？建议先阅读[概念概述](/oss/concepts/context)，了解不同类型的上下文及其使用场景。
</Tip>

### Agent 循环

一个典型的 Agent 循环包含两个主要步骤：

1. **模型调用** - 使用提示词和可用工具调用 LLM，返回响应或请求执行工具
2. **工具执行** - 执行 LLM 请求的工具，返回工具结果

### 你可以控制什么

要构建可靠的 Agent，你需要控制循环中每一步发生的事情，以及步骤之间的行为。

| 上下文类型 | 控制内容 | 临时性或持久性 |
|--------------|------------------|-------------------------|
| **[模型上下文](#model-context)** | 模型调用的输入（指令、消息历史、工具、响应格式） | 临时（Transient） |
| **[工具上下文](#tool-context)** | 工具可以访问和产生的内容（对 State、Store、运行时上下文的读写） | 持久（Persistent） |
| **[生命周期上下文](#life-cycle-context)** | 模型调用和工具调用之间发生的事情（摘要、护栏、日志等） | 持久（Persistent） |

### 数据源

在整个过程中，Agent 会访问（读取/写入）不同的数据源：

| 数据源 | 别名 | 作用域 | 示例 |
|-------------|---------------|-------|----------|
| **运行时上下文（Runtime Context）** | 静态配置 | 会话级 | 用户 ID、API 密钥、数据库连接、权限、环境设置 |
| **状态（State）** | 短期记忆 | 会话级 | 当前消息、上传的文件、认证状态、工具结果 |
| **存储（Store）** | 长期记忆 | 跨会话 | 用户偏好、提取的洞察、记忆、历史数据 |

### 工作原理

LangChain [中间件](/oss/langchain/middleware)是底层机制，它使上下文工程在 LangChain 开发中切实可行。

中间件允许你在 Agent 生命周期的任意步骤中插入钩子，实现以下操作：

* 更新上下文
* 跳转到 Agent 生命周期的其他步骤

在本指南中，你会频繁看到中间件 API 的使用，它是实现上下文工程的核心手段。

### 系统提示词

系统提示词设定了 LLM 的行为和能力。不同的用户、场景或对话阶段可能需要不同的指令。成功的 Agent 会利用记忆、用户偏好和配置信息，为当前对话状态提供正确的指令。

### 消息

消息构成了发送给 LLM 的提示词。
管理消息内容至关重要，这确保了 LLM 拥有正确的信息来给出良好的响应。

<Tabs>
  <Tab title="State">
    当用户查询与已上传文件相关时，从 State 中注入文件上下文：

    :::python

    ```python
    from langchain.agents import create_agent
    from langchain.agents.middleware import wrap_model_call, ModelRequest, ModelResponse
    from typing import Callable

    @wrap_model_call
    def inject_file_context(
        request: ModelRequest,
        handler: Callable[[ModelRequest], ModelResponse]
    ) -> ModelResponse:
        """注入用户在本次会话中上传的文件上下文。"""
        # 从 State 读取：获取已上传文件的元数据
        uploaded_files = request.state.get("uploaded_files", [])  # [!code highlight]

        if uploaded_files:
            # 构建可用文件的上下文
            file_descriptions = []
            for file in uploaded_files:
                file_descriptions.append(
                    f"- {file['name']} ({file['type']}): {file['summary']}"
                )

            file_context = f"""Files you have access to in this conversation:
{chr(10).join(file_descriptions)}

Reference these files when answering questions."""

            # 在近期消息之前注入文件上下文
            messages = [  # [!code highlight]
                *request.messages,
                {"role": "user", "content": file_context},
            ]
            request = request.override(messages=messages)  # [!code highlight]

        return handler(request)

    agent = create_agent(
        model="gpt-4.1",
        tools=[...],
        middleware=[inject_file_context]
    )
    ```
    :::

    :::js
    ```typescript
    import { createMiddleware } from "langchain";

    const injectFileContext = createMiddleware({
      name: "InjectFileContext",
      wrapModelCall: (request, handler) => {
        // request.state 是 request.state.messages 的快捷方式
        const uploadedFiles = request.state.uploadedFiles || [];  // [!code highlight]

        if (uploadedFiles.length > 0) {
          // 构建可用文件的上下文
          const fileDescriptions = uploadedFiles.map(file =>
            `- ${file.name} (${file.type}): ${file.summary}`
          );

          const fileContext = `Files you have access to in this conversation:
${fileDescriptions.join("\n")}

Reference these files when answering questions.`;

          // 在近期消息之前注入文件上下文
          const messages = [  // [!code highlight]
            ...request.messages,  // 其余对话内容
            { role: "user", content: fileContext }
          ];
          request = request.override({ messages });  // [!code highlight]
        }

        return handler(request);
      },
    });

    const agent = createAgent({
      model: "gpt-4.1",
      tools: [...],
      middleware: [injectFileContext],
    });
    ```

    ```python
                {"role": "user", "content": style_context}
            ]
            request = request.override(messages=messages)  # [!code highlight]

        return handler(request)

    agent = create_agent(
        model="gpt-4.1",
        tools=[...],
        middleware=[inject_writing_style],

        store=InMemoryStore()
    ```

    ```python
    @dataclass

    def inject_compliance_rules(

        # 从运行时上下文读取：获取合规要求

            rules.append("- Users have right to data deletion")

        if rules:

{chr(10).join(rules)}"""

                *request.messages,
                {"role": "user", "content": compliance_context}
            ]
            request = request.override(messages=messages)  # [!code highlight]

        return handler(request)

    agent = create_agent(
        model="gpt-4.1",
    ```

<Note>
**临时更新 vs 持久更新：**

上述示例使用 `wrap_model_call` 进行**临时**更新 —— 修改发送给模型的消息，但不会改变 State 中保存的内容。

对于修改 State 的**持久**更新（如[生命周期上下文](#summarization)中的摘要示例），请使用 `before_model` 或 `after_model` 等生命周期钩子来永久更新对话历史。详见[中间件文档](/oss/langchain/middleware)。
</Note>

### 工具

工具让模型能够与数据库、API 和外部系统进行交互。工具的定义和选择方式直接影响模型能否有效完成任务。

#### 定义工具

每个工具都需要清晰的名称、描述、参数名称和参数描述。这些不仅仅是元数据——它们会引导模型推理何时以及如何使用该工具。

:::python
```python
from langchain.tools import tool

@tool(parse_docstring=True)
def search_orders(
    user_id: str,
    status: str,
    limit: int = 10
) -> str:
    """Search for user orders by status.

    Use this when the user asks about order history or wants to check
    order status. Always filter by the provided status.

    Args:
        user_id: Unique identifier for the user
        status: Order status: 'pending', 'shipped', or 'delivered'
        limit: Maximum number of results to return
    """
    # Implementation here
    pass
```
:::

#### 选择工具

并非每个工具都适合所有场景。工具太多可能会压垂模型（上下文过载）并增加错误；太少则限制能力。动态工具选择能根据认证状态、用户权限、功能开关或对话阶段来调整可用工具集。

<Tabs>
  <Tab title="State">
    仅在对话达到特定里程碑后才启用高级工具：

    :::python

    ```python
    from langchain.agents import create_agent
    from langchain.agents.middleware import wrap_model_call, ModelRequest, ModelResponse
    from typing import Callable

    @wrap_model_call
    def state_based_tools(
        request: ModelRequest,
        handler: Callable[[ModelRequest], ModelResponse]
    ) -> ModelResponse:
        """根据对话 State 过滤工具。"""
        # 从 State 读取：检查用户是否已认证
        state = request.state  # [!code highlight]
        is_authenticated = state.get("authenticated", False)  # [!code highlight]
        message_count = len(state["messages"])

        # 仅在认证后启用敏感工具
        if not is_authenticated:
            tools = [t for t in request.tools if t.name.startswith("public_")]
            request = request.override(tools=tools)  # [!code highlight]
        elif message_count < 5:
    ```

### 响应格式

结构化输出将非结构化文本转换为经过验证的结构化数据。当需要提取特定字段或为下游系统返回数据时，自由格式的文本是不够的。

**工作原理：** 当你提供一个 Schema 作为响应格式时，模型的最终响应保证符合该 Schema。Agent 运行模型/工具调用循环，直到模型完成工具调用，然后将最终响应强制转换为指定格式。

    ```typescript
        const environment = request.runtime.context.environment;  // [!code highlight]

        let responseFormat;
        if (userRole === "admin" && environment === "production") {
          responseFormat = adminResponse;  // [!code highlight]
        } else {
          responseFormat = userResponse;  // [!code highlight]
        }

        return handler({ ...request, responseFormat });
      },
    });
    ```

## 工具上下文

工具的特殊之处在于它们既能读取又能写入上下文。

在最基本的情况下，工具执行时接收 LLM 的请求参数并返回工具消息。工具完成其工作并产生结果。

工具还可以为模型获取重要信息，使其能够执行和完成任务。

### 读取

大多数实际工具不仅仅需要 LLM 的参数。它们需要用户 ID 来查询数据库、API 密钥来访问外部服务，或者当前会话状态来做决策。工具通过 State、Store 和运行时上下文来访问这些信息。

<Tabs>
  <Tab title="State">
    从 State 中读取当前会话信息：

    :::python

    ```python
    from langchain.tools import tool, ToolRuntime
    from langchain.agents import create_agent

    @tool
    def check_authentication(
        runtime: ToolRuntime
    ) -> str:
        """检查用户是否已认证。"""
        # 从 State 读取：检查当前认证状态
        current_state = runtime.state
        is_authenticated = current_state.get("authenticated", False)

        if is_authenticated:
            return "User is authenticated"
        else:
            return "User is not authenticated"

    agent = create_agent(
        model="gpt-4.1",
        tools=[check_authentication]
    )
    ```

    :::js


### 写入

工具结果可以用于帮助 Agent 完成给定任务。工具既可以直接向模型返回结果，

    ```python
    def authenticate_user(

        """认证用户并更新 State。"""
        # 执行认证（简化示例）
        if password == "correct":

                update={"authenticated": True},
    ```

    ```python
    from dataclasses import dataclass
    from langchain.tools import tool, ToolRuntime
    from langchain.agents import create_agent
    from langgraph.store.memory import InMemoryStore

    @dataclass
    class Context:
        user_id: str

    @tool
    def save_preference(
        preference_key: str,
        preference_value: str,
        runtime: ToolRuntime[Context]

        user_id = runtime.context.user_id

        store = runtime.store
        existing_prefs = store.get(("preferences",), user_id)

        # 与新偏好合并
        prefs = existing_prefs.value if existing_prefs else {}
        prefs[preference_key] = preference_value

        # 写入 Store：保存更新后的偏好
        store.put(("preferences",), user_id, prefs)

        return f"Saved preference: {preference_key} = {preference_value}"

    agent = create_agent(
        model="gpt-4.1",
        tools=[save_preference],
        context_schema=Context,
        store=InMemoryStore()
    )
    ```

## 生命周期上下文

控制在核心 Agent 步骤**之间**发生的事情 —— 拦截数据流以实现摘要、护栏、日志记录等横切关注点。

正如你在[模型上下文](#model-context)和[工具上下文](#tool-context)中看到的，[中间件](/oss/langchain/middleware)是使上下文工程切实可行的机制。中间件允许你在 Agent 生命周期的任意步骤中插入钩子，实现以下操作：

1. **更新上下文** - 修改 State 和 Store 以持久化变更、更新对话历史或保存洞察
2. **生命周期跳转** - 根据上下文跳转到 Agent 循环的不同步骤（例如，在满足条件时跳过工具执行，或使用修改后的上下文重新调用模型）

### 示例：摘要

最常见的生命周期模式之一是当对话历史过长时自动压缩。与[模型上下文](#messages)中展示的临时消息裁剪不同，摘要会**持久更新 State** —— 永久地用摘要替换旧消息，以供所有后续轮次使用。

LangChain 为此提供了内置中间件：

:::python
```python
from langchain.agents import create_agent
from langchain.agents.middleware import SummarizationMiddleware

agent = create_agent(
    model="gpt-4.1",
    tools=[...],
    middleware=[
        SummarizationMiddleware(
            model="gpt-4.1-mini",
            trigger={"tokens": 4000},
            keep={"messages": 20},
        ),
    ],
)
```
:::

:::js
```typescript
import { createAgent, summarizationMiddleware } from "langchain";

const agent = createAgent({
  model: "gpt-4.1",
  tools: [...],
  middleware: [
    summarizationMiddleware({
      model: "gpt-4.1-mini",
      trigger: { tokens: 4000 },
      keep: { messages: 20 },
    }),
  ],
});
```
:::

当对话超过 token 限制时，`SummarizationMiddleware` 会自动：
1. 使用单独的 LLM 调用对较旧的消息进行摘要
2. 在 State 中用摘要消息永久替换原始消息
3. 保留近期消息以提供上下文

摘要后的对话历史会被永久更新 —— 后续轮次将看到摘要而非原始消息。

<Note>
如需查看内置中间件的完整列表、可用钩子以及如何创建自定义中间件，请参见[中间件文档](/oss/langchain/middleware)。
</Note>

## 最佳实践

1. **从简单开始** - 先使用静态提示词和工具，在需要时再添加动态特性
2. **增量式测试** - 每次只添加一个上下文工程特性
3. **监控性能** - 跟踪模型调用次数、token 使用量和延迟
4. **使用内置中间件** - 充分利用 [`SummarizationMiddleware`](/oss/langchain/middleware#summarization)、[`LLMToolSelectorMiddleware`](/oss/langchain/middleware#llm-tool-selector) 等
5. **记录上下文策略** - 明确说明传递了哪些上下文以及原因
6. **理解临时性与持久性的区别** ：模型上下文的变更是临时的（单次调用），而生命周期上下文的变更会持久化到 State
