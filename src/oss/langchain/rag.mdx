## 概述

LLM 启用的最强大应用之一是复杂的问答（Q&A）聊天机器人。这些应用能够回答关于特定来源信息的问题。它们使用一种称为检索增强生成（Retrieval Augmented Generation，即 [RAG](/oss/langchain/retrieval/)）的技术。

本教程将展示如何基于非结构化文本数据源构建一个简单的问答应用。我们将演示：

1. 一个 RAG [Agent](#rag-agents)，通过简单工具执行搜索。这是一个很好的通用实现方式。
2. 一个两步 RAG [链](#rag-chains)，每次查询只需一次 LLM 调用。这是对简单查询快速有效的方法。

### 概念

我们将涵盖以下概念：

- **索引构建**：从数据源摄取数据并建立索引的流水线。*这通常在单独的过程中完成。*

- **检索与生成**：实际的 RAG 流程，在运行时接收用户查询，从索引中检索相关数据，然后将其传递给模型。

完成数据索引后，我们将使用 [Agent](/oss/langchain/agents) 作为编排框架来实现检索和生成步骤。

<Note>
    本教程的索引部分主要沿用[语义搜索教程](/oss/langchain/knowledge-base)的内容。

    如果你的数据已经可供搜索（即你已有执行搜索的函数），或者你对该教程的内容已经熟悉，请随时跳到[检索与生成](#2-retrieval-and-generation)部分。
</Note>

```python
import bs4
from langchain.agents import AgentState, create_agent
from langchain_community.document_loaders import WebBaseLoader
from langchain.messages import MessageLikeRepresentation
from langchain_text_splitters import RecursiveCharacterTextSplitter

# Load and chunk contents of the blog
loader = WebBaseLoader(
    web_paths=("https://lilianweng.github.io/posts/2023-06-23-agent/",),
    bs_kwargs=dict(
        parse_only=bs4.SoupStrainer(
            class_=("post-content", "post-title", "post-header")
        )
    ),
)
docs = loader.load()

text_splitter = RecursiveCharacterTextSplitter(chunk_size=1000, chunk_overlap=200)
all_splits = text_splitter.split_documents(docs)

# Index chunks
_ = vector_store.add_documents(documents=all_splits)

# Construct a tool for retrieving context
@tool(response_format="content_and_artifact")
def retrieve_context(query: str):
    """Retrieve information to help answer a query."""
    retrieved_docs = vector_store.similarity_search(query, k=2)
    serialized = "\n\n".join(
        (f"Source: {doc.metadata}\nContent: {doc.page_content}")
        for doc in retrieved_docs
    )
    return serialized, retrieved_docs

tools = [retrieve_context]
# If desired, specify custom instructions
prompt = (
    "You have access to a tool that retrieves context from a blog post. "
    "Use the tool to help answer user queries."
)
agent = create_agent(model, tools, system_prompt=prompt)
```

```typescript
import "cheerio";
import { createAgent, tool } from "langchain";
import { CheerioWebBaseLoader } from "@langchain/community/document_loaders/web/cheerio";
import { RecursiveCharacterTextSplitter } from "@langchain/textsplitters";
import * as z from "zod";

// Load and chunk contents of blog
const pTagSelector = "p";
const cheerioLoader = new CheerioWebBaseLoader(
  "https://lilianweng.github.io/posts/2023-06-23-agent/",
  {
    selector: pTagSelector
  }
);

const docs = await cheerioLoader.load();

const splitter = new RecursiveCharacterTextSplitter({
  chunkSize: 1000,
  chunkOverlap: 200
});
const allSplits = await splitter.splitDocuments(docs);

// Index chunks
await vectorStore.addDocuments(allSplits)

// Construct a tool for retrieving context
const retrieveSchema = z.object({ query: z.string() });

const retrieve = tool(
  async ({ query }) => {
    const retrievedDocs = await vectorStore.similaritySearch(query, 2);
    const serialized = retrievedDocs
      .map(
        (doc) => `Source: ${doc.metadata.source}\nContent: ${doc.pageContent}`
      )
      .join("\n");
    return [serialized, retrievedDocs];
  },
  {
    name: "retrieve",
    description: "Retrieve information related to a query.",
    schema: retrieveSchema,
    responseFormat: "content_and_artifact",
  }
);

const agent = createAgent({ model: "gpt-5", tools: [retrieve] });
```

```typescript
let inputMessage = `What is Task Decomposition?`;

let agentInputs = { messages: [{ role: "user", content: inputMessage }] };

for await (const step of await agent.stream(agentInputs, {
  streamMode: "values",
})) {
  const lastMessage = step.messages[step.messages.length - 1];
  prettyPrint(lastMessage);
  console.log("-----\n");
}
```

## 1. 索引构建

<Note>
**本节是[语义搜索教程](/oss/langchain/knowledge-base)内容的简化版本。**

如果你的数据已经建立索引并可供搜索（即你已有执行搜索的函数），或者你已熟悉[文档加载器](/oss/langchain/retrieval#document_loaders)、[嵌入模型](/oss/langchain/retrieval#embedding_models)和[向量存储](/oss/langchain/retrieval#vectorstores)，请随时跳到[检索与生成](/oss/langchain/rag#2-retrieval-and-generation)部分。

</Note>

索引构建通常按以下流程进行：

1. **加载**：首先我们需要加载数据。这通过[文档加载器](/oss/langchain/retrieval#document_loaders)完成。
2. **分割**：[文本分割器](/oss/langchain/retrieval#text_splitters)将大型 `Document` 拆分为更小的块。这对于索引数据和将其传入模型都很有用，因为大块更难搜索，且无法放入模型有限的上下文窗口。
3. **存储**：我们需要一个地方来存储和索引我们的分块，以便日后搜索。这通常使用[向量存储](/oss/langchain/retrieval#vectorstores)和[嵌入模型](/oss/langchain/retrieval#embedding_models)完成。

![index_diagram](/images/rag_indexing.png)

### 加载文档

我们需要先加载博客文章内容。我们可以使用[文档加载器](/oss/langchain/retrieval#document_loaders)，它们从数据源加载数据并返回 @[Document] 对象列表。

:::python
本例中我们使用 [`WebBaseLoader`](/oss/integrations/document_loaders/web_base)，它使用 `urllib` 从网址加载 HTML，并使用 `BeautifulSoup` 解析为文本。我们可以通过 `bs_kwargs` 传递参数给 `BeautifulSoup` 解析器来自定义 HTML 到文本的解析（参见 [BeautifulSoup 文档](https://beautiful-soup-4.readthedocs.io/en/latest/#beautifulsoup)）。本例中只有 class 为 "post-content"、"post-title" 或 "post-header" 的 HTML 标签是相关的，因此我们将移除其他所有标签。

### 分割文档

加载的文档超过 42k 字符，对于许多模型的上下文窗口来说太长了。即使那些可以容纳整篇文章的模型，也可能难以在超长输入中找到信息。

为此，我们将 @[`Document`] 分割为块，用于嵌入和向量存储。这有助于我们在运行时只检索博客文章中最相关的部分。

与[语义搜索教程](/oss/langchain/knowledge-base)一样，我们使用 `RecursiveCharacterTextSplitter`，它会使用常见分隔符（如换行符）递归地分割文档，直到每个块达到合适的大小。这是通用文本场景的推荐文本分割器。

:::python
```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,  # chunk size (characters)
    chunk_overlap=200,  # chunk overlap (characters)
    add_start_index=True,  # track index in original document
)
all_splits = text_splitter.split_documents(docs)

print(f"Split blog post into {len(all_splits)} sub-documents.")
```
```text
Split blog post into 66 sub-documents.
```

**深入了解**

## 2. 检索与生成

RAG 应用通常按以下流程工作：

1. **检索**：给定用户输入，使用[检索器](/oss/langchain/retrieval#retrievers)从存储中检索相关分块。
2. **生成**：[模型](/oss/langchain/models)使用包含问题和检索数据的提示词生成答案。

![retrieval_diagram](/images/rag_retrieval_generation.png)

现在来编写实际的应用逻辑。我们要创建一个简单的应用，接收用户问题，搜索与该问题相关的文档，将检索到的文档和原始问题传递给模型，并返回答案。

我们将演示：

1. 一个 RAG [Agent](#rag-agents)，通过简单工具执行搜索。这是一个很好的通用实现方式。
2. 一个两步 RAG [链](#rag-chains)，每次查询只需一次 LLM 调用。这是对简单查询快速有效的方法。

### RAG Agent

RAG 应用的一种实现方式是一个带有信息检索工具的简单 [Agent](/oss/langchain/agents)。我们可以通过实现一个包装向量存储的[工具](/oss/langchain/tools)来组装一个最小化的 RAG Agent：

:::python
```python
from langchain.tools import tool

@tool(response_format="content_and_artifact")
def retrieve_context(query: str):
    """Retrieve information to help answer a query."""
    retrieved_docs = vector_store.similarity_search(query, k=2)
    serialized = "\n\n".join(
        (f"Source: {doc.metadata}\nContent: {doc.page_content}")
        for doc in retrieved_docs
    )
    return serialized, retrieved_docs
```

<Tip>

这里我们使用 @[tool 装饰器][@tool] 配置工具，将原始文档作为 [artifact](/oss/langchain/messages#param-artifact) 附加到每个 [ToolMessage](/oss/langchain/messages#tool-message)。这让我们可以在应用中访问文档元数据，与发送给模型的字符串表示分开。

</Tip>
:::
:::js
```typescript
import * as z from "zod";
import { tool } from "@langchain/core/tools";

const retrieveSchema = z.object({ query: z.string() });

const retrieve = tool(
  async ({ query }) => {
    const retrievedDocs = await vectorStore.similaritySearch(query, 2);
    const serialized = retrievedDocs
      .map(
        (doc) => `Source: ${doc.metadata.source}\nContent: ${doc.pageContent}`
      )
      .join("\n");
    return [serialized, retrievedDocs];
  },
  {
    name: "retrieve",
    description: "Retrieve information related to a query.",
    schema: retrieveSchema,
    responseFormat: "content_and_artifact",
  }
);
```
<Tip>
    这里我们指定 `responseFormat` 为 `content_and_artifact`，配置工具将原始文档作为 [artifact](/oss/langchain/messages#param-artifact) 附加到每个 [ToolMessage](/oss/langchain/messages#tool-message)。这让我们可以在应用中访问文档元数据，与发送给模型的字符串表示分开。
</Tip>
:::

    ```python
    from typing import Literal

    ```

### RAG 链

在上述 [Agentic RAG](#rag-agents) 实现中，我们允许 LLM 自行决定是否生成[工具调用](/oss/langchain/models#tool-calling)来帮助回答用户查询。这是一个很好的通用方案，但也有一些权衡：

| ✅ 优势                                                                 | ⚠️ 劣势                                                                 |
|-----------------------------------------------------------------------------|----------------------------------------------------------------------------|
| **按需搜索** —— LLM 可以处理问候、追问和简单查询，无需触发不必要的搜索。 | **两次推理调用** —— 执行搜索时，需要一次调用生成查询，另一次生成最终响应。 |
| **上下文感知查询** —— 通过将搜索作为带有 `query` 输入的工具，LLM 可以自行构造融入对话上下文的查询。 | **可控性降低** —— LLM 可能在需要搜索时跳过，或在不需要时执行额外搜索。 |
| **支持多次搜索** —— LLM 可以为单个用户查询执行多次搜索。 |                                                                            |


另一种常见方法是两步链，其中我们始终执行搜索（可能使用原始用户查询）并将结果作为上下文融入单次 LLM 查询。这种方式每次查询只需一次推理调用，以灵活性换取更低的延迟。

在这种方法中，我们不再循环调用模型，而是进行单次传递。

我们可以通过移除工具并将检索步骤融入自定义提示词来实现这个链：

:::python
```python
from langchain.agents.middleware import dynamic_prompt, ModelRequest

@dynamic_prompt
def prompt_with_context(request: ModelRequest) -> str:
    """Inject context into state messages."""
    last_query = request.state["messages"][-1].text
    retrieved_docs = vector_store.similarity_search(last_query)

    docs_content = "\n\n".join(doc.page_content for doc in retrieved_docs)

    system_message = (
        "You are a helpful assistant. Use the following context in your response:"
        f"\n\n{docs_content}"
    )

    return system_message


agent = create_agent(model, tools=[], middleware=[prompt_with_context])
```
:::
:::js
```typescript
import { createAgent, dynamicSystemPromptMiddleware } from "langchain";
import { SystemMessage } from "@langchain/core/messages";

const agent = createAgent({
  model,
  tools: [],
  middleware: [
    dynamicSystemPromptMiddleware(async (state) => {
        const lastQuery = state.messages[state.messages.length - 1].content;

        const retrievedDocs = await vectorStore.similaritySearch(lastQuery, 2);

        const docsContent = retrievedDocs
        .map((doc) => doc.pageContent)
        .join("\n\n");

        // Build system message
        const systemMessage = new SystemMessage(
        `You are a helpful assistant. Use the following context in your response:\n\n${docsContent}`
        );

        // Return system + existing messages
        return [systemMessage, ...state.messages];
    })
  ]
});
```

<Accordion title="返回源文档">

上述 [RAG 链](#rag-chains)将检索到的上下文融入到该次运行的单个系统消息中。

与 [Agentic RAG](#rag-agents) 实现类似，有时我们想在应用状态中包含原始源文档以访问文档元数据。对于两步链，我们可以通过以下方式实现：

1. 在状态中添加一个键来存储检索到的文档
2. 通过 [模型前置钩子](/oss/langchain/agents#pre-model-hook) 添加一个新节点来填充该键（同时注入上下文）。

:::python
```python
from typing import Any
from langchain_core.documents import Document
from langchain.agents.middleware import AgentMiddleware, AgentState


class State(AgentState):
    context: list[Document]


class RetrieveDocumentsMiddleware(AgentMiddleware[State]):
    state_schema = State

    def before_model(self, state: AgentState) -> dict[str, Any] | None:
        last_message = state["messages"][-1]
        retrieved_docs = vector_store.similarity_search(last_message.text)

        docs_content = "\n\n".join(doc.page_content for doc in retrieved_docs)

        augmented_message_content = (
            f"{last_message.text}\n\n"
            "Use the following context to answer the query:\n"
            f"{docs_content}"
        )
        return {
            "messages": [last_message.model_copy(update={"content": augmented_message_content})],
            "context": retrieved_docs,
        }


agent = create_agent(
    model,
    tools=[],
    middleware=[RetrieveDocumentsMiddleware()],
)
```
:::
:::js
```typescript
import { createMiddleware, Document, createAgent } from "langchain";
import { StateSchema, MessagesValue } from "@langchain/langgraph";
import { z } from "zod";

const CustomState = new StateSchema({
  messages: MessagesValue,
  context: z.array(z.custom<Document>()),
});

const retrieveDocumentsMiddleware = createMiddleware({
  stateSchema: CustomState,
  beforeModel: async (state) => {
    const lastMessage = state.messages[state.messages.length - 1].content;
    const retrievedDocs = await vectorStore.similaritySearch(lastMessage, 2);

    const docsContent = retrievedDocs
      .map((doc) => doc.pageContent)
      .join("\n\n");

    const augmentedMessageContent = [
        ...lastMessage.content,
        { type: "text", text: `Use the following context to answer the query:\n\n${docsContent}` }
    ]

    // Below we augment each input message with context, but we could also
    // modify just the system message, as before.
    return {
      messages: [{
        ...lastMessage,
        content: augmentedMessageContent,
      }]
      context: retrievedDocs,
    }
  },
});

const agent = createAgent({
  model,
  tools: [],
  middleware: [retrieveDocumentsMiddleware],
});
```
