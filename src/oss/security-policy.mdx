LangChain 拥有一个庞大的生态系统，与各种外部资源（如本地和远程文件系统、API 和数据库）进行集成。这些集成使开发者能够创建功能丰富的应用，将 LLM 的能力与访问、交互和操作外部资源的能力相结合。

## 最佳实践

在构建此类应用时，开发者应始终遵循良好的安全实践：

* [**最小权限原则**](https://en.wikipedia.org/wiki/Principle_of_least_privilege)：将权限严格限定在应用的实际需求范围内。授予过广或过大的权限可能引入重大安全漏洞。为避免此类漏洞，建议根据应用实际情况采取相应措施，如使用只读凭据、禁止访问敏感资源、使用沙箱技术（如在容器中运行）、指定代理配置来控制外部请求等。
* **预判潜在滥用**：就像人会犯错一样，大语言模型（LLM）同样会出错。始终假设任何系统访问权或凭据都可能被以其权限允许的任何方式使用。例如，如果一组数据库凭据允许删除数据，最安全的做法是假设任何能使用这些凭据的 LLM 确实可能会删除数据。
* [**纵深防御**](https://en.wikipedia.org/wiki/Defense_in_depth_(computing))：没有任何单一的安全技术是完美的。微调和良好的链设计可以降低但不能消除 LLM 犯错的概率。最好结合多层安全方法，而不是依赖单一防线。例如：同时使用只读权限和沙箱隔离，确保 LLM 只能访问明确允许它们使用的数据。

不遵循上述实践可能导致的风险包括但不限于：

* 数据损坏或丢失。
* 未授权访问机密信息。
* 关键资源的性能下降或不可用。

示例场景及缓解策略：

* 用户可能要求具有文件系统访问权的 Agent 删除不应删除的文件或读取包含敏感信息的文件。缓解方法：将 Agent 限制在特定目录中，仅允许读写安全的文件。可进一步通过在容器中运行来对 Agent 进行沙箱隔离。
* 用户可能要求具有外部 API 写入权限的 Agent 向 API 写入恶意数据或删除 API 数据。缓解方法：给 Agent 只读的 API 密钥，或将其限制为仅使用已具备防护能力的端点。
* 用户可能要求具有数据库访问权的 Agent 删除表或修改 schema。缓解方法：将凭据权限限定为仅允许 Agent 访问的表，并考虑使用只读凭据。

如果你正在构建访问外部资源（如文件系统、API 或数据库）的应用，建议与你公司的安全团队沟通，确定如何以最佳方式设计和保护你的应用。

## 报告开源项目漏洞

请按以下流程报告与 LangChain 开源项目相关的安全漏洞：

1. 在漏洞所在的 GitHub 仓库的 Security 标签页中**提交安全公告**。
2. 发送**邮件**至 `security@langchain.dev`，通知我们你已提交安全问题及其所在仓库。

在报告漏洞之前，请先查看上述[最佳实践](#最佳实践)，了解我们如何界定安全漏洞和开发者责任。

### 漏洞赏金资格

我们欢迎所有 LangChain 库的安全漏洞报告。但我们可能仅对以下包中的漏洞提供临时漏洞赏金：

* 由 LangChain 团队拥有和维护的核心库：`langchain-core`、`langchain`（v1）、`langgraph` 及相关的 checkpointer 包（或其 JavaScript 等价包）
* 由 LangChain 团队维护的流行集成（例如，`langchain-openai`、`langchain-anthropic` 等，或其 JavaScript 等价包）

漏洞必须存在于库代码本身，而非示例代码或示例应用中。

我们欢迎其他所有 LangChain 包的报告，并将解决有效的安全问题，但不会为上述范围之外的包提供漏洞赏金。这包括 `langchain-community`，由于其社区驱动的特性，不符合漏洞赏金资格，但我们仍会接受并处理相关报告。

### 不在范围内的目标

以下内容不在安全漏洞报告范围内：

* **langchain-experimental**：此仓库用于实验性代码，不在安全报告范围内（参见[包警告](https://pypi.org/project/langchain-experimental/)）。
* **示例和示例应用**：示例代码和演示应用不在安全报告范围内。
* **已标注安全声明的代码**：这将根据具体情况决定，但很可能不在范围内，因为代码已附带开发者应遵循的安全指南。
* **LangSmith 相关仓库或 API**：请参见下方[报告 LangSmith 漏洞](#报告-langsmith-漏洞)。
