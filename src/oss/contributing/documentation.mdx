我们欢迎对 LangChain 文档的贡献，包括新功能介绍、[集成文档](/oss/contributing/publish-langchain#adding-documentation)，以及对现有文档的改进。

## 快速开始 - 本地开发

要在本地预览文档：

```bash
git clone https://github.com/langchain-ai/docs.git
```

```bash
cd docs
```

```bash
make install
```

```bash
make dev
```

这将启动一个支持热重载的开发服务器，地址为 `http://localhost:3000`。编辑 `src/` 目录下的文件即可立即看到变更。

<Tip>
如果本地预览遇到问题，请尝试运行 `mint update` 以确保使用的是最新版本的 Mintlify。
</Tip>

<Accordion title="前置条件">
    **必需：**
    - Python 3.13+
    - [uv](https://docs.astral.sh/uv/) - Python 包管理器
    - [Node.js](https://nodejs.org/en) 和 npm
    - [Make](https://www.gnu.org/software/make/)
    - [Git](https://git-scm.com/)

    **可选：**
    - [markdownlint-cli](https://github.com/igorshubovych/markdownlint-cli) - `npm install -g markdownlint-cli`
    - [Mintlify MDX VSCode 扩展](https://www.mintlify.com/blog/mdx-vscode-extension)
</Accordion>

## 编辑文档

<Accordion title="在 GitHub 上快速编辑">
    对于拼写错误或小幅修改，可以直接在 GitHub 上编辑，无需本地环境：

    1. 点击任意页面底部的 **Edit this page on GitHub**。
    1. Fork 到你的个人账户。
    1. 在 GitHub 的 Web 编辑器中进行修改。
    1. 创建 Pull Request。
</Accordion>

<Note>
    **仅编辑 `src/` 目录下的文件**—— `build/` 目录是自动生成的。
</Note>

1. 按照我们的 [写作规范](#写作规范) 编辑 `src/` 目录下的文件。
1. 提交前运行 [质量检查](#运行质量检查)。
1. 创建 Pull Request 等待审核。

<Accordion title="创建可分享的预览构建（仅限 LangChain 团队）">
    当你创建或更新 PR 时，会自动生成一个 [预览分支/ID](https://github.com/langchain-ai/docs/actions/workflows/create-preview-branch.yml)。PR 上会有一条评论包含该 ID。

    1. 从评论中复制预览分支的 ID
    2. 在 [Mintlify 控制台](https://dashboard.mintlify.com/langchain-5e9cc07a/langchain-5e9cc07a?section=previews) 中，点击 **Create preview deployment**
    3. 输入预览分支的 ID，点击 **Create deployment**
    4. 选择预览并点击 **Visit** 查看

    要使用最新更改重新部署，请在控制台点击 **Redeploy**。
</Accordion>

### 运行质量检查

提交更改前，请确保代码通过格式化和 lint 检查：

```bash
# 检查失效链接
make mint-broken-links

# 自动格式化代码
make format

# 检查 lint 问题
make lint

# 修复 Markdown 问题
make lint_md_fix

# 运行测试确保你的更改不会破坏现有功能
make test
```

更多详细信息，请参阅 `README` 中的 [可用命令](https://github.com/langchain-ai/docs?tab=readme-ov-file#available-commands) 部分。

<Important>
    所有 Pull Request 都会由 CI/CD 自动检查。相同的 lint 和格式化标准将被强制执行，未通过这些检查的 PR 无法合并。
</Important>

所有文档分为以下四类：

    </Card>

    <Card title="概念指南" icon="bulb" href="#概念指南">
        提供更深层次理解和洞见的解释性内容。

    </Card>

    </Card>

<Note>

### 操作指南

操作指南是面向知道自己想完成什么任务的用户的操作说明。操作指南的示例可在 [LangChain](/oss/langchain/overview) 和 [LangGraph](/oss/langgraph/overview) 标签页中找到。

<AccordionGroup>
    <Accordion title="特点">
        - **聚焦任务**：聚焦于特定的任务或问题
        - **分步进行**：将任务分解为较小的步骤
        - **实践导向**：提供具体的示例和代码片段
    </Accordion>

    <Accordion title="编写建议">
        - 侧重于**"怎么做"**而非"为什么"
        - 使用具体的示例和代码片段
        - 将任务分解为较小的步骤
        - 链接到相关的概念指南和参考文档
    </Accordion>

    <Accordion title="示例">
        - [Messages](/oss/langchain/messages)
        - [Tools](/oss/langchain/tools)
        - [Streaming](/oss/langgraph/streaming)
    </Accordion>
</AccordionGroup>

### 概念指南

概念指南以抽象的方式介绍核心概念，帮助读者深入理解。

<AccordionGroup>
    <Accordion title="特点">
        - **理解导向**：解释事物为何如此运作
        - **视角宏观**：比其他类型的文档有更高、更广的视角
        - **面向设计**：解释设计决策和权衡取舍
        - **上下文丰富**：使用类比和对比
    </Accordion>

    <Accordion title="编写建议">
        - 侧重于**"为什么"**而非"怎么做"
        - 提供功能使用所不必需的补充信息
        - 可以使用类比并引用替代方案
        - 避免混入过多参考文档内容
        - 链接到相关的教程和操作指南
    </Accordion>

    <Accordion title="示例">
        - [Memory](/oss/concepts/memory)
        - [Context](/oss/concepts/context)
        - [Graph API](/oss/langgraph/graph-api)
        - [Functional API](/oss/langgraph/functional-api)
    </Accordion>
</AccordionGroup>

### API 参考

API 参考文档包含详细的底层信息，精确描述已有的功能及其使用方法。

<CardGroup cols={2}>
    <Card title="Python 参考" href="https://reference.langchain.com/python/" icon="brand-python" arrow />
    <Card title="JavaScript/TypeScript 参考" href="https://reference.langchain.com/javascript/" icon="brand-javascript" arrow />
</CardGroup>

好的参考文档应该：
- 描述所有已有功能（所有参数、选项、返回值）
- 全面且结构化，便于查阅
- 作为技术细节的权威来源

<AccordionGroup>
    <Accordion title="如何贡献参考文档">
        :::python
        参阅 [Python 参考文档](https://github.com/langchain-ai/docs/blob/main/reference/python/README.md) 的贡献指南。
        :::
        :::js
        参阅 [JavaScript/TypeScript 参考文档](https://github.com/langchain-ai/docs/blob/main/reference/javascript/README.md) 的贡献指南。
        :::
    </Accordion>

    <Accordion title="LangChain 参考文档最佳实践">
        - **保持一致**：遵循现有的 Provider 文档模式
        - 包含基本用法（代码片段）和常见的边缘情况/失败场景
        - 注明需要特定版本的功能
    </Accordion>

    <Accordion title="何时创建新的参考文档">
        - 新的集成或 Provider 需要专门的参考页面
        - 复杂的配置选项需要详细说明
        - API 变更引入了新的参数或行为
        - 社区经常就特定功能提出问题
    </Accordion>
</AccordionGroup>

### 教程

教程是较长篇幅的分步指南，层层递进，引导用户通过特定的实践活动来构建理解。教程通常位于 [Learn](/oss/learn) 标签页中。

<Note>
    我们通常不会合并外部贡献者提交的新教程，除非有明确的需求。如果你认为某个主题在文档中缺失或覆盖不够充分，请 [提交新的 Issue](https://github.com/langchain-ai/docs/issues)。
</Note>

<AccordionGroup>
    <Accordion title="特点">
        - **实践性**：聚焦于实践活动以构建理解
        - **分步进行**：将活动分解为较小的步骤
        - **动手操作**：提供顺序连贯、可运行的代码片段
        - **补充性**：提供功能使用所不必需的额外上下文和信息
    </Accordion>

    <Accordion title="编写建议">
        - 代码片段应是顺序连贯的，用户按步骤操作即可运行
        - 为活动提供一些上下文背景，但将更详细的信息链接到相关的概念指南和参考文档
    </Accordion>

    <Accordion title="示例">
        - [语义搜索](/oss/langchain/knowledge-base)
        - [RAG Agent](/oss/langchain/rag)
    </Accordion>
</AccordionGroup>

### Mintlify 组件

使用 [Mintlify 组件](https://mintlify.com/docs/text) 提升可读性：

<Tabs>
    <Tab title="提示框">
    - `<Note>` 用于有用的补充信息
    - `<Warning>` 用于重要警告和破坏性变更
    - `<Tip>` 用于最佳实践和建议
    - `<Info>` 用于中性的上下文信息
    - `<Check>` 用于成功确认
</Tab>

<Tab title="结构">
    - `<Steps>` 用于顺序操作的概览。**不要**用于长列表式步骤或教程。
    - `<Tabs>` 用于特定平台的内容。
    - `<AccordionGroup>` 和 `<Accordion>` 用于非必需但有价值的信息，默认折叠（例如完整代码示例）。
    - `<CardGroup>` 和 `<Card>` 用于突出展示内容。
</Tab>

<Tab title="代码">
    - `<CodeGroup>` 用于多语言示例。
    - 始终在代码块上指定语言标签（例如 ` ```python`、` ```javascript`）。
    - 为代码块添加标题（例如 `Success`、`Error Response`）。
    </Tab>

### Python 与 JavaScript/TypeScript 内容并置

所有文档在可能的情况下都必须同时提供 Python 和 JavaScript/TypeScript 两个版本。为此，我们使用自定义的内联语法来区分应在一种或两种语言版本中显示的内容段落：

```mdx
\:::python
Python 专属内容。在实际文档中，`python` 前面的反斜杠应省略。
\:::

\:::js
JavaScript/TypeScript 专属内容。在实际文档中，`js` 前面的反斜杠应省略。
\:::

两种语言通用的内容（不用包裹）
```

这将生成两个输出文件（每种语言一个），路径为 `/oss/python/concepts/foo.mdx` 和 `/oss/javascript/concepts/foo.mdx`。每个输出页面都需要添加到 `/src/docs.json` 文件中才能出现在导航中。

<Note>
    我们不希望语言版本的不对等阻碍贡献。如果某个功能目前仅在一种语言中可用，可以暂时只提供该语言的文档，直到另一种语言跟上。在这种情况下，请加入说明，指出该功能在另一种语言中尚不可用。

    如果你需要在 Python 和 JavaScript/TypeScript 之间翻译内容，请在 [社区 Slack](https://www.langchain.com/join-community) 中求助或在 PR 中 @ 维护者。
</Note>

## 质量标准

### 通用准则

<AccordionGroup>
    <Accordion title="避免重复">
        多个页面涵盖相同内容会增加维护难度并造成混淆。每个概念或功能应只有一个权威页面。使用链接引用其他指南，而不是重新解释。
    </Accordion>

    <Accordion title="频繁使用链接">
        文档各部分不是孤立存在的。频繁链接到其他部分，帮助用户了解不熟悉的主题。这包括链接到 API 参考和概念部分。
    </Accordion>

    <Accordion title="保持简洁">
        采用"少即是多"的原则。如果其他部分已有好的解释，链接过去而不是重新解释，除非你的内容提供了新的角度。
    </Accordion>
</AccordionGroup>

### 无障碍要求

确保文档对所有用户都是可访问的：

- 使用标题和列表来组织内容，便于快速浏览
- 使用具体、可操作的链接文本，而不是"点击这里"
- 为所有图片和图表提供描述性的 alt 文本

### 交叉引用

使用一致的交叉引用将文档与 API 参考文档连接起来。

**从文档链接到 API 参考：**

使用 `@[]` 语法链接到 API 参考页面：

```mdx
See \@[`ChatAnthropic`] for all configuration options.

The \@[`bind_tools`][ChatAnthropic.bind_tools] method accepts...
```

构建流水线会根据当前的语言作用域（Python 或 JavaScript）将这些转换为正确的 Markdown 链接。例如，`\@[ChatAnthropic]` 会根据正在构建的文档版本链接到 Python 或 JS 的 API 参考页面，**但前提是在 `link_map.py` 文件中存在对应条目！** 详见下文。

<Accordion title="自动链接的工作原理">

`@[]` 语法由 [`handle_auto_links.py`](https://github.com/langchain-ai/docs/blob/main/pipeline/preprocessors/handle_auto_links.py) 处理。它在 [`link_map.py`](https://github.com/langchain-ai/docs/blob/main/pipeline/preprocessors/link_map.py) 中查找链接键，该文件包含 Python 和 JavaScript 两个作用域的字典映射。

**支持的格式：**

| 语法 | 结果 |
|--------|--------|
| `\@[ChatAnthropic]` | 以"ChatAnthropic"作为显示文本的链接 |
| `` \@[`ChatAnthropic`] `` | 以 `` `ChatAnthropic` ``（代码格式）作为显示文本的链接 |
| `\@[text][ChatAnthropic]` | 以"text"作为显示文本，以 `ChatAnthropic` 作为链接映射中键的链接 |
| `\\@[ChatAnthropic]` | 转义：渲染为字面文本 `\@[ChatAnthropic]`（不生成链接——本页面中使用的就是这种方式！） |

**添加新链接：**

如果在映射中未找到链接，输出中将保持原样不变。要添加新的自动链接：

1. 打开 `pipeline/preprocessors/link_map.py`
2. 在 `LINK_MAPS` 中适当的作用域（`python` 或 `js`）下添加条目
3. 键是 `\@[key]` 或 `\@[text][key]` 中使用的链接名称，值是相对于参考文档主机的路径

</Accordion>

:::python
**从 API 参考 stub 链接到 OSS 文档：**

参阅 [`README`](https://github.com/langchain-ai/docs/blob/main/reference/python/README.md) 了解更多关于从 API 参考 stub 链接到 Python OSS 文档的信息。特别是 `mkdocstrings` 的交叉引用 [链接语法](https://github.com/langchain-ai/docs/blob/main/reference/python/README.md#mkdocsmkdocstrings-python-cross-reference-linking-syntax)。
:::
