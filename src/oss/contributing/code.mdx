### 快速修复：提交 Bug 修复

对于简单的 Bug 修复，你可以立即开始：

<Steps>
    <Step title="复现问题">
        在克隆仓库之前，请确保你能可靠地复现该 Bug。这有助于确认问题并为你的修复提供起点。维护者和其他贡献者应该能够根据你的描述复现问题，无需额外的设置或修改。
    </Step>

    <Step title="Fork 仓库">
        :::python
        将 [LangChain](https://github.com/langchain-ai/langchain)、[LangGraph](https://github.com/langchain-ai/langgraph) 或 [Deep Agents](https://github.com/langchain-ai/deepagents) 仓库 Fork 到你的<Tooltip tip="如果 Fork 到组织账户，维护者将无法进行编辑，这可能会显著延迟接受时间。">个人 GitHub 账户</Tooltip>
        :::
        :::js
        将 [LangChain](https://github.com/langchain-ai/langchainjs)、[LangGraph](https://github.com/langchain-ai/langgraphjs) 或 [Deep Agents](https://github.com/langchain-ai/deepagentsjs) 仓库 Fork 到你的<Tooltip tip="如果 Fork 到组织账户，维护者将无法进行编辑，这可能会显著延迟接受时间">个人 GitHub 账户</Tooltip>
        :::
    </Step>

    <Step title="克隆并设置环境">
        :::python

        ```bash
        git clone https://github.com/your-username/name-of-forked-repo.git

        # 例如，对于 LangChain：
        git clone https://github.com/parrot123/langchain.git
        ```

        ```bash
        # 在仓库目录中，初始化环境并安装依赖
        uv venv && source .venv/bin/activate
        uv sync --all-groups

        # 或者，只安装特定的依赖组：
        uv sync --group test
        ```

        如果你之前尚未安装 [`uv`](https://docs.astral.sh/uv/)，需要先安装它
        :::
        :::js

        ```bash
        git clone https://github.com/your-username/name-of-forked-repo.git

        # 例如，对于 LangChain：
        git clone https://github.com/parrot123/langchainjs.git

        # 对于 LangGraph：
        git clone https://github.com/parrot123/langgraphjs.git
        ```

        ```bash
        # 在仓库目录中，安装依赖
        pnpm install
        # 为所有包创建构建以解析工作区依赖
        pnpm build
        ```
        :::
    </Step>

    <Step title="创建分支">
        为你的修复创建一个新分支。这有助于保持代码组织有序，并方便后续提交 Pull Request。

        ```bash
        git checkout -b your-username/short-bugfix-name
        ```
    </Step>

    <Step title="编写失败测试">
        添加在没有你的修复时会失败的 [单元测试](#测试编写指南)。这使我们能够验证 Bug 已修复并防止回归
    </Step>

    <Step title="进行修改">
        在遵循我们的 [代码质量标准](#代码质量标准) 的前提下修复 Bug。进行**必要的最小改动**来解决问题。我们强烈建议贡献者在开始编码之前先在 Issue 上留言。例如：

        > *"我想处理这个问题。我计划的方法是 [……简要描述……]。这是否符合维护者的预期？"*

        30 秒的留言往往能避免因初始方向错误而浪费大量精力。
    </Step>

    :::js
    <Step title="运行构建">
        运行构建命令以确保包仍能正常构建
        ```bash
        pnpm build
        # 或构建特定的工作区包
        pnpm --filter @langchain/core build
        ```
    </Step>
    :::

    <Step title="验证修复">
        确保测试通过且未引入回归问题。提交 PR 前确保所有测试在本地通过

        :::python

        ```bash
        make format
        make lint
        make test

        # 对于涉及集成的 Bug 修复，还需运行：
        make integration_tests
        # （可能需要设置 API 测试凭据）
        ```
        :::
        :::js

        ```bash
        pnpm lint
        pnpm test

        # 对于涉及集成的 Bug 修复，还需运行：
        pnpm test:int

        # 或在特定工作区包中运行测试
        cd libs/langchain-core
        pnpm test
        pnpm lint

        # 或从仓库根目录运行特定包的测试
        pnpm --filter @langchain/core test
        pnpm --filter @langchain/core lint
        ```
        :::
    </Step>

    <Step title="记录变更">
        如果行为发生变化，请更新文档字符串和/或行内注释
    </Step>

    <Step title="提交 Pull Request">
        按照提供的 PR 模板进行操作。如适用，使用 [关闭关键字](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/linking-a-pull-request-to-an-issue#linking-a-pull-request-to-an-issue-using-a-keyword) 引用你正在修复的 Issue（例如 `Fixes #ISSUE_NUMBER`），这样当 PR 合并时 Issue 会自动关闭。
    </Step>
</Steps>

## 贡献准则

在开始为 LangChain 项目做贡献之前，请先思考一下你的动机。如果你的唯一目标是在简历上添加"首次贡献"（或者只是寻找一个快速的成就），那么参加训练营或在线教程可能更合适。

为开源项目做贡献需要时间和精力，但它也能帮助你成为更好的开发者并学习新技能。不过要注意，这可能比跟着培训课程学习更难、更慢。话虽如此，如果你愿意花时间把事情做好，为开源做贡献是非常值得的！

### 向后兼容性

<Warning>
    除了关键的安全修复外，不允许对公共 API 进行破坏性变更。

    有关主版本发布的详细信息，请参阅我们的 [版本策略](/oss/versioning)。
</Warning>

通过以下方式保持兼容性：

<AccordionGroup>
    <Accordion title="稳定接口">
        **始终保留**：

        - 函数签名和参数名称
        - 类接口和方法名称
        - 返回值的结构和类型
        - 公共 API 的导入路径
    </Accordion>

    <Accordion title="安全变更">
        **可接受的修改**：

        :::python
        - 添加新的可选参数
        :::
        :::js
        - 添加新的可选参数/类型参数
        :::
        - 为类添加新方法
        - 在不改变行为的前提下提升性能
        - 添加新的模块或函数
    </Accordion>

    <Accordion title="修改前的检查">
        - **这会破坏现有的用户代码吗？**
        - 检查你的修改目标是否是公共 API
        :::python
        - 如果需要，它是否在 `__init__.py` 中导出了？
        :::
        - 测试中是否存在现有的使用模式？
    </Accordion>
</AccordionGroup>

### 新功能

我们对新功能的标准要求较高。一般来说，我们不接受外部贡献者提交的新核心抽象，除非有相关 Issue 证明存在迫切需求。这同样适用于基础设施和依赖项的变更。

通常，功能贡献的要求包括：

<Steps>
    <Step title="设计讨论">
        创建 Issue 描述：

        - 你要解决的问题
        - 提议的 API 设计
        - 预期的使用模式
    </Step>

    <Step title="实现">
        - 遵循现有的代码模式
        - 包含全面的测试和文档
        - 考虑安全影响
    </Step>

    <Step title="集成考量">
        - 这与现有功能如何交互？
        - 是否存在性能影响？
        - 是否引入了新的依赖？

        我们将拒绝可能导致安全漏洞或安全报告的功能。
    </Step>
</Steps>

### 安全准则

<Warning>
    安全至关重要。切勿引入漏洞或不安全的模式。
</Warning>

安全检查清单：

<AccordionGroup>
    <Accordion title="输入验证">
        - 验证和清理所有用户输入
        - 正确转义模板和查询中的数据
        :::python
        - 切勿对用户数据使用 `eval()`、`exec()` 或 `pickle`，因为这可能导致任意代码执行漏洞
        :::
        :::js
        - 切勿使用 `eval()`，因为这可能导致任意代码执行漏洞
        :::
    </Accordion>

    <Accordion title="错误处理">
        - 使用具体的异常类型
        - 不要在错误信息中暴露敏感信息
        - 实施适当的资源清理
    </Accordion>

    <Accordion title="依赖管理">
        - 避免添加硬依赖
        - 保持可选依赖最少化
        - 审查第三方包的安全问题
    </Accordion>
</AccordionGroup>

## 开发工作流

### 运行测试

<Info>
    以下目录路径相对于你正在开发的包。
</Info>

在可能的情况下，我们倾向于使用单元测试而非集成测试。单元测试在每个 Pull Request 上都会运行，因此应该快速且可靠。集成测试按计划运行，需要更多设置，因此应仅用于确认与外部服务的接口是否正常。

#### 单元测试

:::python
**位置**：`tests/unit_tests/`
:::
:::js
**位置**：`src/tests/FILENAME_BEING_TESTED.test.ts`
:::

单元测试覆盖不需要调用外部 API 的模块化逻辑。如果你添加了新的逻辑，应该添加相应的单元测试。在单元测试中，检查预处理/后处理并 mock 外部依赖。

**要求**：
- 不允许网络调用
- 测试所有代码路径，包括边缘情况
- 使用 mock 替代外部依赖

运行单元测试：

:::python
```bash
make test

# 或直接运行：
uv run --group test pytest tests/unit_tests

# 运行特定测试文件：
TEST_FILE=tests/unit_tests/test_imports.py make test
```
:::
:::js
```bash
# 运行完整测试套件
pnpm test

# 或运行特定测试文件
pnpm test src/tests/FILENAME_BEING_TESTED.test.ts

# 或运行特定测试函数
pnpm test -t "the test that should be run"
```
:::

#### 集成测试

:::python
**位置**：`tests/integration_tests/`
:::
:::js
**位置**：`src/tests/FILENAME_BEING_TESTED.int.test.ts`
:::

集成测试覆盖需要调用外部 API（通常是与其他服务的集成）的逻辑。

集成测试需要访问外部服务/Provider API（可能产生费用），因此默认不运行。

并非每次代码修改都需要集成测试，但请注意，作为审核流程的一部分，我们会单独要求/运行集成测试。

**要求**：
- 测试与外部服务的真实集成
- 使用环境变量存储 API 密钥
- 在凭据不可用时优雅跳过

运行集成测试：
:::python
```bash
make integration_tests

# 或直接运行：
uv run --group test --group test_integration pytest --retries 3 --retry-delay 1 tests/integration_tests

# 运行特定测试文件：
TEST_FILE=tests/integration_tests/test_openai.py make integration_tests
```
:::
:::js
```bash
pnpm test:int
```
:::

### 代码质量标准

贡献必须遵守以下质量要求：

<Tabs>
    <Tab title="类型标注">
        :::python
        **必需**：所有函数都需要完整的类型标注

        ```python
        def process_documents(
            docs: list[Document],
            processor: DocumentProcessor,
            *,
            batch_size: int = 100
        ) -> ProcessingResult:
            """Process documents in batches.

            Args:
                docs: List of documents to process.
                processor: Document processing instance.
                batch_size: Number of documents per batch.

            Returns:
                Processing results with success/failure counts.
            """
        ```
        :::

### 依赖管理

LangChain 包区分**硬依赖**和**可选依赖**，以保持包的轻量并减少用户的安装开销。

<Tabs>
    <Tab title="可选依赖">
        几乎所有新依赖都应该是可选的。在以下情况使用可选依赖：

        - 该依赖仅用于特定集成或功能
        - 用户在不使用该依赖的情况下仍可以正常使用包
        - 该依赖体积较大或有大量传递依赖

        **要求：**

        - 未安装该依赖的用户必须能够**导入**你的代码而不产生任何副作用（无警告、无错误、无异常）
        - `pyproject.toml` 和 `uv.lock` **不应**被修改

        **添加可选依赖的步骤：**

        1. 将依赖添加到相应的测试依赖文件中（例如 `extended_testing_deps.txt`）
        2. 添加至少尝试导入新代码的单元测试。理想情况下，单元测试使用轻量级 fixture 来测试代码逻辑
        3. 对需要该依赖的单元测试使用 `@pytest.mark.requires("package_name")` 装饰器
    </Tab>

    <Tab title="硬依赖">
        硬依赖在用户安装包时自动安装。仅在以下情况使用硬依赖：

        - 包的核心功能完全离不开该依赖
        - 该依赖体积小且传递依赖少
        - 没有合理的方式将该功能设为可选

        <Warning>
            添加硬依赖会增加所有用户的安装时间并可能引发版本冲突。

            维护者将仔细审查硬依赖的添加！
        </Warning>

        **添加硬依赖的步骤：**

        1. 创建 Issue 或发起讨论，说明为什么该依赖必须是硬依赖而非可选依赖
        2. 将依赖添加到 `pyproject.toml` 的相应部分
        3. 运行 `uv lock` 更新 lock 文件
        4. 包含覆盖新功能的全面测试
    </Tab>
</Tabs>
:::

### 测试编写指南

为了编写有效的测试，需要遵循一些良好实践：

:::python
- 在文档字符串中使用自然语言描述测试
- 使用描述性变量名
- 穷尽所有断言

:::js
- 将测试封装在描述被测组件的 `describe` 块中
- 使用自然语言描述测试名称
- 穷尽所有断言
- 仅对合理大小的数据对象使用快照

        def test_document_processor_handles_empty_input():
