[记忆](/oss/langgraph/add-memory)是一个能够记住先前交互信息的系统。对于 AI Agent 而言，记忆至关重要，因为它让 Agent 能够记住先前的交互、从反馈中学习、并适应用户偏好。随着 Agent 处理的任务越来越复杂、用户交互越来越多，这种能力对于效率和用户满意度都变得不可或缺。

本概念指南介绍两类记忆，按其召回范围分类：

* [短期记忆](#short-term-memory)，即[线程](/oss/langgraph/persistence#threads)级别的记忆，通过在会话内维护消息历史来跟踪正在进行的对话。LangGraph 将短期记忆作为 Agent [state](/oss/langgraph/graph-api#state) 的一部分来管理。State 通过 [checkpointer](/oss/langgraph/persistence#checkpoints) 持久化到数据库，使线程可以在任何时候恢复。短期记忆在图被调用或某个步骤完成时更新，State 在每个步骤开始时被读取。
* [长期记忆](#long-term-memory)存储用户特定或应用级别的数据，跨会话共享，并能跨对话线程访问。它可以在_任何时间_、_任何线程_中被召回。记忆的作用域可以是任意自定义命名空间，而不仅仅是单个线程 ID。LangGraph 提供了 [stores](/oss/langgraph/persistence#memory-store)（[参考文档](https://langchain-ai.github.io/langgraph/reference/store/#langgraph.store.base.BaseStore)），让你能够保存和召回长期记忆。

![Short vs long](/oss/images/short-vs-long.png)

## 短期记忆

[短期记忆](/oss/langgraph/add-memory#add-short-term-memory)使你的应用能够记住单个[线程](/oss/langgraph/persistence#threads)或对话中的先前交互。一个[线程](/oss/langgraph/persistence#threads)组织了一次会话中的多次交互，类似于邮件将消息分组到同一个对话中的方式。

LangGraph 将短期记忆作为 Agent state 的一部分来管理，通过线程级别的 checkpoint 进行持久化。这个 state 通常包含对话历史以及其他有状态的数据，如上传的文件、检索的文档或生成的产物。通过将这些存储在图的 state 中，机器人可以访问给定对话的完整上下文，同时保持不同线程之间的隔离。

### 管理短期记忆

对话历史是短期记忆中最常见的形式，而很长的对话对当前的 LLM 来说是一个挑战。完整的历史可能无法装进 LLM 的上下文窗口，从而导致不可恢复的错误。即使你的 LLM 支持完整的上下文长度，大多数 LLM 在超长上下文上的表现仍然不佳。它们会被过时或离题的内容“分散注意力”，同时还会面临响应速度变慢和成本升高的问题。

聊天模型通过消息来接受上下文，消息包括开发者提供的指令（系统消息）和用户输入（人类消息）。在聊天应用中，消息在人类输入和模型响应之间交替，形成一个随时间不断增长的消息列表。由于上下文窗口有限，而 token 密集的消息列表开销很高，许多应用可以从手动移除或遗忘过时信息的技术中获益。

![Filter](/oss/images/filter.png)

关于管理消息的常用技术，详见[添加和管理记忆](/oss/langgraph/add-memory#manage-short-term-memory)指南。

## 长期记忆

LangGraph 的[长期记忆](/oss/langgraph/add-memory#add-long-term-memory)允许系统在不同对话或会话之间保留信息。与作用域限于**线程**的短期记忆不同，长期记忆保存在自定义的“命名空间”中。

长期记忆是一个复杂的挑战，没有一套适用于所有场景的解决方案。不过，以下问题可以为你提供一个导航不同技术的框架：

* 记忆的类型是什么？人类使用记忆来记住事实（[语义记忆](#semantic-memory)）、经历（[情景记忆](#episodic-memory)）和规则（[程序记忆](#procedural-memory)）。AI Agent 也可以以相同的方式使用记忆。例如，AI Agent 可以使用记忆来记住关于用户的特定事实，以便完成任务。
* [何时更新记忆？](#writing-memories)记忆可以作为 Agent 应用逻辑的一部分进行更新（即“在热路径上”）。在这种情况下，Agent 通常会在响应用户之前先决定要记住哪些事实。或者，记忆也可以作为后台任务更新（在后台/异步运行的逻辑生成记忆）。我们在[下文](#writing-memories)中详细解释这些方法的权衡取舍。

不同的应用需要不同类型的记忆。虽然类比并不完美，但研究[人类记忆类型](https://www.psychologytoday.com/us/basics/memory/types-of-memory?ref=blog.langchain.dev)可以带来有益的启发。一些研究（如 [CoALA 论文](https://arxiv.org/pdf/2309.02427)）甚至将这些人类记忆类型映射到了 AI Agent 中使用的记忆类型。

| 记忆类型 | 存储内容 | 人类示例 | Agent 示例 |
|-------------|----------------|---------------|---------------|
| [语义记忆](#semantic-memory) | 事实 | 在学校学到的知识 | 关于用户的事实 |
| [情景记忆](#episodic-memory) | 经历 | 做过的事情 | Agent 过去的操作 |
| [程序记忆](#procedural-memory) | 指令 | 本能或运动技能 | Agent 的系统提示词 |

### 语义记忆

无论是在人类还是 AI Agent 中，[语义记忆](https://en.wikipedia.org/wiki/Semantic_memory)都涉及特定事实和概念的保留。在人类中，它可以包括在学校学到的信息以及对概念及其关系的理解。对于 AI Agent，语义记忆常通过记住过去交互中的事实或概念来实现应用的个性化。

<Note>
语义记忆与“语义搜索”不同。语义搜索是一种通过“语义”（通常用向量嵌入表示）查找相似内容的技术。语义记忆是心理学术语，指的是存储事实和知识；而语义搜索是一种基于意义而非精确匹配来检索信息的方法。
</Note>

语义记忆可以通过不同的方式进行管理：

#### 档案模式

记忆可以是一个单一的、持续更新的“档案”，包含关于用户、组织或其他实体（包括 Agent 本身）的明确、特定的信息。档案通常就是一个 JSON 文档，包含你为业务领域选定的各种键值对。

在记忆档案时，你需要确保每次都是在**更新**档案。因此，你需要传入之前的档案，并[让模型生成新的档案](https://github.com/langchain-ai/memory-template)（或某种 [JSON 补丁](https://github.com/hinthornw/trustcall)来应用到旧档案上）。随着档案变大，这种方式可能会容易出错，可以考虑将档案拆分为多个文档，或在生成文档时使用**严格模式**解码，以确保记忆 schema 保持有效。

![Update profile](/oss/images/update-profile.png)

#### 集合模式

另一种方式是将记忆存储为一个文档集合，随时间不断更新和扩展。每条单独的记忆作用域更窄、更容易生成，这意味着你不太可能随时间推移而**丢失**信息。对于 LLM 来说，为新信息生成_新_对象比将新信息与现有档案进行调和要容易得多。因此，文档集合往往能带来更高的[召回率](https://en.wikipedia.org/wiki/Precision_and_recall)。

然而，这种方式将一些复杂性转移到了记忆更新环节。模型现在需要能够_删除_或_更新_列表中的现有条目，这可能比较棘手。此外，某些模型可能倾向于过度插入，另一些可能倾向于过度更新。可以参考 [Trustcall](https://github.com/hinthornw/trustcall) 包来管理这个问题，并考虑使用评估工具（如 [LangSmith](/langsmith/evaluation)）来调优行为。

使用文档集合还会将复杂性转移到列表上的记忆**检索**环节。`Store` 目前支持[语义搜索](https://langchain-ai.github.io/langgraph/reference/store/#langgraph.store.base.SearchOp.query)和[内容过滤](https://langchain-ai.github.io/langgraph/reference/store/#langgraph.store.base.SearchOp.filter)两种方式。

最后，使用记忆集合可能使得为模型提供全面的上下文变得困难。虽然单条记忆可能遵循特定的 schema，但这种结构可能无法捕捉记忆之间的完整上下文或关联关系。因此，在使用这些记忆生成响应时，模型可能缺少重要的上下文信息，而这些信息在统一档案方法中会更容易获取。

![Update list](/oss/images/update-list.png)

无论采用哪种记忆管理方式，核心要点是 Agent 会使用语义记忆来为其响应提供[事实基础](/oss/langchain/retrieval)，这通常能带来更个性化和相关的交互体验。

### 情景记忆

无论是在人类还是 AI Agent 中，[情景记忆](https://en.wikipedia.org/wiki/Episodic_memory)都涉及对过去事件或操作的回忆。[CoALA 论文](https://arxiv.org/pdf/2309.02427)对此有很好的阐述：事实可以写入语义记忆，而*经历*可以写入情景记忆。对于 AI Agent，情景记忆通常用于帮助 Agent 记住如何完成某项任务。

### 程序记忆

无论是在人类还是 AI Agent 中，[程序记忆](https://en.wikipedia.org/wiki/Procedural_memory)都涉及记住执行任务所用的规则。在人类中，程序记忆类似于执行任务的内化知识，比如通过基本运动技能和平衡感来骑自行车。而情景记忆则是回忆特定的经历，比如第一次不用辅助轮成功骑车或一次难忘的景区骑行。对于 AI Agent，程序记忆是模型权重、Agent 代码和 Agent 提示词的组合，共同决定了 Agent 的功能。

在实践中，Agent 修改自己的模型权重或重写代码的情况相当少见。但 Agent 修改自己的提示词则更为常见。

一种有效的 Agent 指令优化方法是通过[“反思”](https://blog.langchain.dev/reflection-agents/)或元提示（meta-prompting）。具体做法是将 Agent 当前的指令（如系统提示词）连同最近的对话或明确的用户反馈一起传给 Agent，然后让 Agent 根据这些输入优化自己的指令。这种方法特别适合难以预先指定指令的任务，因为它允许 Agent 从交互中学习和调整。

例如，我们构建了一个 [Tweet 生成器](https://www.youtube.com/watch?v=Vn8A3BxfplE)，利用外部反馈和提示词重写来为 Twitter 生成高质量的论文摘要。在这种场景下，具体的摘要提示词很难*预先*指定，但用户可以很容易地对生成的 Tweet 提出批评意见，并提供关于如何改进摘要过程的反馈。

以下伪代码展示了如何使用 LangGraph 记忆 [store](/oss/langgraph/persistence#memory-store) 来实现这一功能：使用 store 保存提示词，`update_instructions` 节点获取当前提示词（以及从与用户对话中捕获到的反馈，存储在 `state["messages"]` 中），更新提示词，然后将新提示词保存回 store。接着，`call_model` 从 store 获取更新后的提示词，并用它生成响应。

:::python
```python
# Node that *uses* the instructions
def call_model(state: State, store: BaseStore):
    namespace = ("agent_instructions", )
    instructions = store.get(namespace, key="agent_a")[0]
    # Application logic
    prompt = prompt_template.format(instructions=instructions.value["instructions"])
    ...

# Node that updates instructions
def update_instructions(state: State, store: BaseStore):
    namespace = ("instructions",)
    instructions = store.search(namespace)[0]
    # Memory logic
    prompt = prompt_template.format(instructions=instructions.value["instructions"], conversation=state["messages"])
    output = llm.invoke(prompt)
    new_instructions = output['new_instructions']
    store.put(("agent_instructions",), "agent_a", {"instructions": new_instructions})
    ...
```
:::

:::js
```typescript
// Node that *uses* the instructions
const callModel = async (state: State, store: BaseStore) => {
    const namespace = ["agent_instructions"];
    const instructions = await store.get(namespace, "agent_a");
    // Application logic
    const prompt = promptTemplate.format({
        instructions: instructions[0].value.instructions
    });
    // ...
};

// Node that updates instructions
const updateInstructions = async (state: State, store: BaseStore) => {
    const namespace = ["instructions"];
    const currentInstructions = await store.search(namespace);
    // Memory logic
    const prompt = promptTemplate.format({
        instructions: currentInstructions[0].value.instructions,
        conversation: state.messages
    });
    const output = await llm.invoke(prompt);
    const newInstructions = output.new_instructions;
    await store.put(["agent_instructions"], "agent_a", {
        instructions: newInstructions
    });
    // ...
};
```
:::

### 写入记忆

Agent 写入记忆主要有两种方法：[“在热路径上”](#in-the-hot-path)和[“在后台”](#in-the-background)。

![Hot path vs background](/oss/images/hot_path_vs_background.png)

#### 在热路径上

在运行时创建记忆既有优势也有挑战。优势在于，这种方法允许实时更新，新记忆可以立即用于后续交互。它还支持透明性，用户可以在记忆被创建和存储时收到通知。

然而，这种方法也存在挑战。如果 Agent 需要一个新工具来决定要将什么写入记忆，可能会增加复杂性。此外，关于保存什么的推理过程可能会影响 Agent 的延迟。最后，Agent 必须在创建记忆和其他职责之间多任务并行，这可能会影响创建的记忆数量和质量。

例如，ChatGPT 使用 [save_memories](https://openai.com/index/memory-and-new-controls-for-chatgpt/) 工具以内容字符串的形式插入或更新记忆，在每条用户消息时决定是否以及如何使用此工具。参见我们的 [memory-agent](https://github.com/langchain-ai/memory-agent) 模板作为参考实现。

#### 在后台

将记忆创建作为单独的后台任务有多个优势。它消除了主应用的延迟影响，将应用逻辑与记忆管理分离，并允许 Agent 更专注地完成任务。这种方法还提供了灵活地安排记忆创建时机以避免冗余工作的优势。

然而，这种方法也有自己的挑战。确定记忆写入的频率变得至关重要，因为更新不及时可能导致其他线程缺少新的上下文。决定何时触发记忆生成也很重要。常见策略包括：在设定时间段后调度（如果有新事件则重新调度）、使用定时调度（cron），或允许用户或应用逻辑手动触发。

参见我们的 [memory-service](https://github.com/langchain-ai/memory-template) 模板作为参考实现。

### 记忆存储

LangGraph 将长期记忆以 JSON 文档的形式存储在 [store](/oss/langgraph/persistence#memory-store) 中。每条记忆被组织在一个自定义的 `namespace`（类似文件夹）和一个唯一的 `key`（类似文件名）下。命名空间通常包含用户 ID、组织 ID 或其他标签，便于组织信息。这种结构支持记忆的层次化组织，并通过内容过滤器支持跨命名空间的检索。

:::python
```python
from langgraph.store.memory import InMemoryStore


def embed(texts: list[str]) -> list[list[float]]:
    # Replace with an actual embedding function or LangChain embeddings object
    return [[1.0, 2.0] * len(texts)]


# InMemoryStore saves data to an in-memory dictionary. Use a DB-backed store in production use.
store = InMemoryStore(index={"embed": embed, "dims": 2})
user_id = "my-user"
application_context = "chitchat"
namespace = (user_id, application_context)
store.put(
    namespace,
    "a-memory",
    {
        "rules": [
            "User likes short, direct language",
            "User only speaks English & python",
        ],
        "my-key": "my-value",
    },
)
# get the "memory" by ID
item = store.get(namespace, "a-memory")
# search for "memories" within this namespace, filtering on content equivalence, sorted by vector similarity
items = store.search(
    namespace, filter={"my-key": "my-value"}, query="language preferences"
)
```
:::

:::js
```typescript
import { InMemoryStore } from "@langchain/langgraph";

const embed = (texts: string[]): number[][] => {
    // Replace with an actual embedding function or LangChain embeddings object
    return texts.map(() => [1.0, 2.0]);
};

// InMemoryStore saves data to an in-memory dictionary. Use a DB-backed store in production use.
const store = new InMemoryStore({ index: { embed, dims: 2 } });
const userId = "my-user";
const applicationContext = "chitchat";
const namespace = [userId, applicationContext];

await store.put(
    namespace,
    "a-memory",
    {
        rules: [
            "User likes short, direct language",
            "User only speaks English & TypeScript",
        ],
        "my-key": "my-value",
    }
);

// get the "memory" by ID
const item = await store.get(namespace, "a-memory");

// search for "memories" within this namespace, filtering on content equivalence, sorted by vector similarity
const items = await store.search(
    namespace,
    {
        filter: { "my-key": "my-value" },
        query: "language preferences"
    }
);
```
:::
